\section{Introduction}
Dans le contexte actuel de la cybersécurité, les menaces internes et externes sur les infrastructures serveurs sont omniprésentes. L'importance des journaux (logs) s'est imposée comme une nécessité absolue pour tracer chaque événement sur le système. Cependant, les logs ne sont utiles que s'ils sont correctement analysés et surveillés. L'objectif de ce projet est de déployer un système complet de surveillance et de logs de sécurité sur un OS Linux, capable de détecter les élévations de privilèges, les modifications de fichiers sensibles et les échecs de connexions SSH via une approche automatisée.

\section{État de l’art}
Il existe plusieurs outils de journalisation sur Linux. \texttt{syslog} ou \texttt{rsyslog} permettent une gestion centralisée des logs applicatifs ou d'authentification (\texttt{auth.log}). \texttt{journald} gère les logs via systemd. Cependant, pour une approche orientée sécurité "au cœur" du noyau, \texttt{auditd} (Linux Audit Daemon) est la référence incontournable. Il permet de scruter les appels systèmes. En outre, des SIEM (Security Information and Event Management) légers ou lourds (comme Wazuh, Splunk, ou ELK) agrègent souvent ces données, bien qu'une détection sur l'hôte via des scripts locaux Python ou Bash soit plus adaptée pour un système modeste.

\section{Problématique}
Pourquoi les logs par défaut d'un système Linux ne suffisent-ils pas ?
Bien que le système génère des journaux, il n'existe aucune alerte automatisée sans une configuration avancée. Sans une politique d'audit précise :
\begin{itemize}
    \item L'accès et la lecture d'un fichier sensible (comme \texttt{/etc/shadow}) ne sont pas tracés.
    \item Les tentatives de Force Brute ou de mouvements latéraux peuvent noyer les journaux sans alerte pour l'administrateur.
    \item La création persistante d'un attaquant (nouvel utilisateur ou compte compromis) peut passer inaperçue.
\end{itemize}

\section{Architecture de la Solution}
Notre plateforme de test repose sur deux machines virtuelles dans un environnement de réseau local.
\begin{itemize}
    \item \textbf{VM1 (Serveur Cible Ubuntu, OS à sécuriser)} : IP \texttt{192.168.56.10}. Elle intègre \texttt{auditd}, \texttt{rsyslog}, des scripts d'analyse \texttt{bash} et une interface Python (\texttt{Flask}) pour l'affichage visuel en direct.
    \item \textbf{VM2 (Kali Linux ou Ubuntu de test)} : IP \texttt{192.168.56.20}. Elle simule l'attaquant (Brute force via SSH, etc.).
\end{itemize}

\section{Implémentation et Outils Développés}
\subsection{Configuration des Règles}
Nous avons implémenté des règles \texttt{auditd} sur la VM1 afin de traquer précisément :
\begin{itemize}
    \item Les changements de privilèges (exécution de \texttt{sudo} ou \texttt{su}).
    \item Les modifications de \texttt{/etc/passwd} et \texttt{/etc/shadow}.
    \item Les opérations relatives aux binaires contenant des informations confidentielles temporelles ou d'audit.
\end{itemize}
\subsection{Développement du Tableau de Bord (Dashboard) Python}
Afin de ne pas se limiter à une recherche manuelle avec \texttt{ausearch}, nous avons développé une application graphique sous Python utilisant Flask. Elle expose une interface Web pour parser de manière dynamique et continue (via rafraîchissement) le fichier \texttt{/var/log/auth.log} et les résultats formatés d'\texttt{auditd}.

\section{Expérimentation}
Nous avons défini 4 scénarios d'attaques pour valider le système :
\begin{enumerate}
    \item \textbf{Brute Force SSH} (Tentatives répétées avec un faux compte utilisateur ou mot de passe).
    \item \textbf{Usage malveillant de Sudo} (Élévation).
    \item \textbf{Modification fichier sensible} (Simulation par la commande \texttt{touch /etc/passwd}).
    \item \textbf{Changement de mot de passe persistant} (\texttt{passwd}).
\end{enumerate}

\section{Limites et Perspectives}
La principale limite de notre système est la sur-verbosité et le risque de "faux positifs" pour des utilisateurs normaux d'outils d'administration sur de très grands clusters, générant de la "fatigue d'alerte". La lecture continue (\texttt{tail} direct ou parsing brutal de fichiers) via le back-end Flask peut aussi montrer des dysfonctionnements en cas de \texttt{logrotate}.
En perspectives, ce système peut être intégré dans une architecture ELK complète ou envoyant un e-mail à l'administrateur système pour les événements jugés hautement critiques (via Python \texttt{smtplib}).

\section{Conclusion}
Le projet a prouvé de façon concrète la pertinence d'intégrer des outils en couche profonde noyau comme \texttt{auditd} et des front-end visuels développés en Python ou Bash pour alerter un administrateur sur les principales failles d'escalade d'un noyau fonctionnant sur environnement GNU/Linux. L'implémentation mise en place permet une réactivité rapide contre les intrusions.
